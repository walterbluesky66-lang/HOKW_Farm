# 王者荣耀世界家园种菜助手

本项目是一个本地网页小工具，用来帮助玩家管理《王者荣耀世界》家园种田和牧场。玩家在游戏中手动种植作物或养殖动物后，在网页中打卡；如果已经进行了一段时间，也可以按游戏显示的剩余时间同步接管。网页根据作物类型、自浇、好友浇水、动物催产收获时间和真实经过时间，自动计算下一次该做什么。

## 当前形态

- 静态网页，直接在浏览器中运行；Cloudflare Pages 发布时仍只发布 `dist/`。
- 作物标准库固化在 `app.js` 随网站发布；其余用户数据默认保存在 `localStorage`，配置 Supabase 后可用邮箱验证码登录并自动同步私有云存档，云端旧快照迁移不会触发反复自动刷新。
- 当前管理对象是一整块菜地的一批作物，不是多块独立菜地。
- 牧场按栏位总数管理动物数量，用户只填写 16 小时动物和 20 小时动物各多少只，并按两组批次计时。
- 暂不管理药物作物，也不把药物作物混入经验或百工币收益计算。
- 支持浏览器通知、页面内弹窗和柔和提示音。
- 作物浇水提醒按“最大间隔优先、最佳收尾更早则切换收尾”的策略触发，不再按最早可自浇时间提醒，减少不必要的浇水次数。
- 支持同步已有作物：在主页面打卡面板内切换到“同步已有”，填写剩余成熟时间、当前自浇可减时间，或距离下次可浇等待时间。
- 支持作物/动物档案库二级页面：作物资料为只读标准库，动物资料仍可录入和编辑，并用于收益对比和牧场自动选动物。
- 支持作物/动物性价比计算器二级页面：作物按“满级基准单价 × 产量”比较经验、基准百工币和孙膑周期收益，不叠菜摊；动物比较每轮净收益、经验效率和回本周期。
- 支持牧场管理二级页面：设置栏位数、动物摊等级、16 小时动物数量和 20 小时动物数量，并按两组催产收获时间提醒。
- 支持居所设置二级页面：维护当前居所等级、农田数量和菜摊等级；菜摊等级仅用于作物档案参考售价展示。
- 支持一键规划二级页面：按当前居所等级、农田数量、周末目标和睡眠时间，先展示“现在该做什么”、总体时间进度和当前作物进度；完整 7 天排期放在三级详情页。
- 支持用户与存档管理二级页面：集中处理 Supabase 云存档登录、本地备份导出和 JSON 导入。
- 支持兴趣圈经验记录二级页面：按日期录入每组 TAG 的累计经验，自动计算今日新增、离 206 差值和 8200 达标剩余天数。

## 作物类型

- 3分钟测试作物：仅用于测试，不做成熟时间向上取整。
- 5小时快速作物。
- 16小时经验作物。
- 20小时经济作物。
- 28小时经验作物。

## 核心规则

设 `T` 为作物原始成熟时间。

- 首次自浇：种下后立即可浇，减少 `7%T`。
- 后续自浇：
  - 最小浇水间隔为 `6%T`。
  - 最大浇水间隔为 `24%T`。
  - 距离上次自浇时间小于 `6%T` 时，不能浇水或减时为 0。
  - 在 `6%T` 到 `24%T` 区间内，减时从 `1.75%T` 线性增加到 `7%T`。
  - 超过 `24%T` 后，最多仍只减少 `7%T`。
- 好友浇水：每次减少 `2.5%T`，最多 4 次，总上限 `10%T`。

## 成熟与进度

有效成长进度按下面公式计算：

```text
有效成长进度 = 当前时间 - 种植时间 + 所有浇水减少时间
```

当有效成长进度大于或等于 `T` 时，作物成熟。

进度条显示的是有效成长进度，不是单纯倒计时百分比。因此自然时间流逝会推进进度，首次浇水、好友浇水、后续自浇也会让进度条瞬间增长对应比例。

真实作物 `5h/16h/20h/28h` 的成熟时间会向上取整到分钟；3 分钟测试作物不会向上取整，方便验证倒计时、提醒和历史 BUG。

当前作物卡片顶部、农作物标题右侧提供“我浇水了 / 好友浇水 +1 / 孙膑催熟 / 收获完成”等常用互动按钮，方便在看到作物状态时直接打卡。

## 同步已有作物

如果玩家打开工具前已经种田，可以在主页面“整块菜地种植打卡”里切换到“同步已有”，然后填写：

- 作物成熟时间。
- 游戏显示的剩余成熟时间。
- 当前自浇状态：现在浇水可以减少多少时间，或距离下一次可浇还要等多久。
- 可选：是否已用孙膑催熟、已发生好友浇水次数。

同步会校准当前倒计时和未来浇水节点。历史自浇、好友浇水减时不会被反推；已发生好友浇水次数只用于限制后续还能记录几次，不会重复减少成熟时间。

## 牧场管理

牧场数据单独保存在 `wzry-world-farm-ranch-v1`。玩家可以在 `ranch.html` 设置：

- 当前牧场栏位数：1-12 个。
- 动物摊等级：每级让动物产物售价提升 5%，例如 Lv.27 为基础售价的 2.3 倍，Lv.28 为 2.35 倍。
- 16 小时动物数量和 20 小时动物数量，二者总和会自动限制在栏位数以内。
- 每组动物游戏显示的剩余收获时间。

动物会自动采用当前居所等级可用的同成熟时间最高等级档案；同等级下，16 小时动物优先经验，20 小时动物优先每轮净收益。动物到达原始成熟时间的 `3/4` 时即可催产收获，所以 16 小时动物按 12 小时提醒，20 小时动物按 15 小时提醒。点击“我收获了”后，该组动物会开始下一轮完整催产计时。

主页会显示牧场提醒卡片，包括最近收获批次、当前可收动物数量和养殖数量。卡片顶部提供“收获”批次按钮，玩家不用进入二级页面也能直接记录对应 16h / 20h 批次的收获，并让该批次开始下一轮完整催产计时。浏览器通知、页面内提示和提示音会同时覆盖作物和动物。

## 作物/动物档案库与收益对比

“作物/动物档案库”二级页面用于查看和维护具体资料。作物档案已改为只读标准库，来自居所图鉴并固化在 `app.js` 中；旧的用户自录作物不会再被读取、导出、导入或云同步。作物档案包括：

- 农作物名称。
- 解锁居所等级。
- 成熟时间。
- 产量。
- 种子成本。
- 1 级基准单价和满级基准单价。
- 单块农田基准百工币收益：`满级基准单价 × 产量`。
- 可获得经验。

Lv.1-Lv.5 作物只在档案中展示，不进入作物收益对比、一键规划、当前作物自动选档或任何收益/排期计算。高于当前居所等级的作物默认折叠，可点击查看；当前可用作物按等级从高到低展示。居所设置中的菜摊等级只用于档案里的“菜摊参考”收益展示，不影响计算器和规划。

动物档案包括：

- 动物名称和所需居所等级。
- 购买价格、回收价格；回收价格留空时默认购买价格的一半。
- 成熟时间档位：16 小时或 20 小时。
- 每轮收获预估饲料价格，默认 3200。
- 满亲密度动物产物出售总价，按未计入动物摊的基础售价录入。
- 每轮经验。

收益对比不再手动输入名称和收益，而是在“作物/动物性价比计算器”二级页面中从档案库选择条目加入分析。作物下拉只显示当前居所等级可种的 Lv.6 及以上标准库作物；作物百工币收益、百工币效率和孙膑周期收益均按基准收益计算，不叠菜摊。动物档案被编辑后，当前动物收益对比会自动读取最新数据。

动物计算口径：

```text
动物摊倍率 = 1 + (动物摊等级 - 1) × 5%
产物售价 = 满亲密度产物基础售价 × 动物摊倍率
每轮净百工币 = 产物售价 - 饲料费
回本成本 = 购买价格 - 回收价格
回本周期 = 回本成本 / 每轮净百工币
换养额外净投入 = 目标动物回本成本 - 基准动物回本成本
换养追平时间 = 换养额外净投入 / 单位时间净收益差
```

如果每轮净百工币小于或等于 0，回本周期显示为“无法回本”。动物经验效率按催产收获时间换算；同周期经验差值会和用户选择的基准动物比较。换养追平会用同一个基准动物比较净投入差和单位时间净收益差；相同催产周期时，也会折算为追平轮数和完整收获轮数。

动物档案页录入的数据会保存在当前浏览器 `localStorage`。如果要把动物档案做成发给别人也自带的标准库，可以用页面里的“导出动物标准库”生成数据，再固化到 `app.js` 的 `BUILT_IN_ANIMAL_ARCHIVE` 数组中。作物标准库只能通过维护 `BUILT_IN_CROP_ARCHIVE` 后重新发布来更新。

## 一键规划

“一键规划”二级页面会读取只读标准作物库，根据当前居所等级筛选 Lv.6 及以上可种作物，再按当前时间生成 7 天排期。主页只保留入口，不直接展开完整排期。二级页面优先展示下一步建议、总体时间进度和当前作物进度；完整一周规划在 `planner-detail.html` 三级页面中用箭头卡片查看。

当前牧场不纳入一键规划排期；牧场只负责批次计时、提醒和动物收益计算。

- 主页种植或同步已有作物时，选择某个成熟时间档位后，会自动绑定当前等级可用的同档位最高等级标准作物；同等级有多条时，经验作物优先经验，20小时作物优先基准百工币。
- 普通收获期：周一 00:00 到周五 08:00 前，优先规划经验收益。
- 双倍收获期：周五 08:00 到下周一 00:00，目标未达成时只按当前等级可用的最高等级 20 小时高百工币作物推进目标；收获该作物时按双倍作物数量计算，经验不翻倍。
- 周五 08:00 前种下、但预计在 08:00 后收获的作物，会按双倍窗口参与规划。
- 周末 20 小时高百工币作物目标默认 6000 个，可在规划页修改；一块田普通按 30 个、双倍按 60 个计算。
- 规划页可选择是否计入好友祈福。勾选时会把好友祈福成功的三倍产量纳入目标辅助判断：双倍窗口内每次成功祈福按额外 `60 × (3 - 1) = 120` 个目标数量计算，并按目标作物覆盖天数估算每天约 2-5 次成功祈福；如果若干轮目标作物加上可行祈福次数已经能完成目标，后续不再额外追加高价值目标作物。取消勾选时，周末目标只按基础双倍收获计算。
- 目标达成后，后续排期转为经验优先。
- 睡眠时间内不安排自浇，且规划会尽量避免作物在睡眠时间成熟。
- 规划排期按理论最短成熟时间推演，理论口径包含首次自浇、后续自浇和 4 次好友浇水；再结合睡眠避让决定实际可操作收获时间。
- 当前正在种的作物也会按理论最短成熟时间占用排期，并在二级页单独展示当前作物进度条；进度条展示自浇和收尾自浇等可预期节点，好友浇水不标具体时间，只作为理论最短成熟假设写入备注。
- 规划模型会滚动枚举“种某个作物”和“等待到关键窗口”两类动作，动态比较 7 天窗口内的目标数量、双倍百工币、经验收益和睡眠空等；因此会判断睡眠前是先补短周期、提前种长周期，还是等待双倍窗口更划算。
- 规划不会自动覆盖当前作物。点击“按规划种下”后，才会把推荐作物写入当前作物管理卡片。
- 当前作物浇水、好友浇水、孙膑、收获或用户延误后，规划都会按最新状态重新计算。
- 二级页总体进度条只展示种植与收获轮换，不展示好友浇水时间；周末目标进度在顶部用圆环展示作物获取数量。三级页按“作物卡片 → 作物卡片”的箭头链展示完整一周计划。

## 兴趣圈经验记录

“兴趣圈”二级页面用于核对每日兴趣圈活动经验。它只在主页顶部提供入口，核心逻辑放在 `interest-circle.html` 和 `interest-circle.js`，后续可以相对容易地移除或单独使用。

- 默认内置 8 组 TAG，并把整理表中的两天历史值作为初始记录：`2026-06-20` 对应“前一天经验”，`2026-06-21` 对应“今天经验”。
- 页面打开时自动选中当天日期；用户可以用上一天、下一天或日历查看和修改历史记录。
- 页面录入的是每组 TAG 的“累计经验值”，不是今日新增值。
- 今日新增 = 当前日期累计经验 - 该 TAG 组最近一个历史记录日的累计经验。
- 离满额差值 = 今日新增 - 206；满额显示 `0`，少拿显示负数，超过 206 会显示正数并提示核对。
- 8200 达标剩余天数按最新可用累计经验和每天满额 `206` 推算。
- TAG 分类可新增；已有分类在各自卡片内单独编辑分类名称和对应 TAG，不提供整类删除入口，只能删除分类下的单个 TAG，以保留历史经验记录。

## 用户与存档管理

顶部“用户与存档管理”入口会打开二级页面，集中支持两种方式：

- 未登录或未配置 Supabase 时，继续使用本地浏览器数据，并可手动导出/导入 JSON。
- 配置 Supabase 后，用户用邮箱验证码登录；登录后读取 `primary` 云端快照，后续本地数据变化会延迟几秒自动上传。

云端快照沿用本地备份 JSON 结构，包含：

- 种植打卡。
- 动物档案库。
- 作物/动物收益对比。
- 牧场批次。
- 一键规划设置与进度。
- 兴趣圈经验记录。

浏览器通知权限和已提醒标记不会导出，也不会进入云端快照。云端恢复前会在当前浏览器保存一份非同步恢复副本，避免误覆盖后完全找不回本地数据。

## 理论最快与最佳收尾浇水

作物卡片里的“理论最短成熟”按当前状态继续推演，不再从种下那一刻重新计算。同步已有作物后，它会以游戏剩余成熟时间为基准，继续估算剩余好友浇水和后续自浇能带来的最快成熟时间。

“最佳收尾浇水”用于提示最后一次自浇的推荐时间点。理论口径是：前面按最大浇水间隔拿较高减时，最后一次自浇尽量卡在“浇完后作物刚好成熟”的时间点，避免过早浇水导致减时不足，也避免已经能成熟后才补记录。

主页顶部“剩余成熟”卡片会同时展示普通预计成熟时间和理论最短成熟时间。原“下次自浇”顶部卡片已改为“距离最大间隔”：通常提示等到最大浇水间隔再浇；如果最佳收尾浇水时间早于最大间隔，则这张卡片会直接显示最佳收尾时间。浏览器通知、页面内提示和提示音也按这个目标时间触发，而不是按最早可自浇时间触发。

## 本地启动

直接用浏览器打开 `index.html` 即可。

## 在线部署

在线发布使用 Cloudflare Pages：

```powershell
npm run build
```

构建后会生成 `dist/` 目录，只包含网页运行必需文件。Cloudflare Pages 配置建议：

- Build command：`npm run build`
- Build output directory：`dist`
- Production branch：`main`

不要把项目根目录直接作为公网目录发布，因为根目录包含 `AGENTS.md`、`README.md`、`TODO.md`、Excel 源文件和原始参考 HTML。`dist/` 已在 `.gitignore` 中忽略，部署平台会在每次发布时重新生成。

要启用云存档，还需要在 Cloudflare Pages 的环境变量中配置：

- `HOKW_SUPABASE_URL`
- `HOKW_SUPABASE_PUBLISHABLE_KEY`

`scripts/build-site.js` 会把这两个值写入 `dist/cloud-config.js`。前端只使用 Supabase publishable key；不要把 service role 或 secret key 放进仓库或前端环境变量。

## Supabase 配置

1. 创建 Supabase 项目，开启 Authentication 的 Email provider。
2. 如果希望输入数字验证码，在邮件模板中使用 `{{ .Token }}`；如果使用默认 magic link，当前页面的验证码输入不会完成登录。
3. 打开 Supabase SQL Editor，执行 `supabase/user_snapshots.sql`。
4. 在 Cloudflare Pages 项目里添加 `HOKW_SUPABASE_URL` 和 `HOKW_SUPABASE_PUBLISHABLE_KEY`，然后重新部署。
5. 用两个不同邮箱验证：A 账号只能恢复 A 的存档，不能读写 B 的存档。
